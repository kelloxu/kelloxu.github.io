Spark系列三——1-弹性分布式数据集

John Xu
Spark系列三

发布于：2020年10月22日
次浏览

Spark系列二中介绍了Spark的结构化API，在绝大多数场景下使用这些结构化API就可以解决业务问题，但当这类高级API无法解决特殊的业务或工程问题时，就需要使用Spark的低级API，特别是弹性分布式数据集（RDD）、SparkContext和分布式共享变量（例如累加器和广播变量）。

1 什么是低级API

有两种低级API：

  • 一种用于处理分布式数据（RDD）
  • 另一种用于分发和处理分布式共享变量（广播变量和累加器）

1.1 何时使用低级API

  • 当在高级API中找不到所需功能时（例如要对集群中数据的物理放置进行非常严格的控制时）
  • 当需要维护一些使用RDD编写的遗留代码库时
  • 当需要执行一些自定义共享变量操作时（关于共享变量，请查看 Spark系列三——3-分布式共享变量）

当调用一个DataFrame的转换操作时，实际上等价于一组RDD的转换操作，当你开始条调试越来越复杂的工作负载，这种对等价转换的理解可以帮你更容易完成任务。

1.2 如何使用低级API

SparkContext 是低级API函数库的入口，可以通过 SparkSession 来获取 SparkContext，SparkSession 是用于在 Spark 集群上执行计算的工具（将在下一篇博客中介绍SparkSeesion）。

2. 关于RDD

RDD 是 Spark1.x 中主要的API，在 Spark 2.x 中仍然可以使用它，但已不常用，因为主要使用 DataFrame 和 Dataset，不过，无论是 DataFrame 还是 Dataset，运行的所有 Spark 代码都将编译成一个 RDD，所以对 RDD是什么以及如何使用它要有一个基本的了解。

RDD 是一个只读不可变的且且已分块的记录集合，并可以被并行处理。

RDD 与 DataFrame 不同，DataFrame中每个记录是一个结构化的数据行，各字段已知且 schema 已知，而 RDD 中的记录仅仅是程序员选择的 Java、Scala或Pythond对象，正因为如此，我们可以完全控制 RDD，即能以任何格式在这些对象中存储任何内容，但是这也带来了一些潜在问题：

  • 无论实现什么任务，都必须从底层开发
  • Spark不象对结构化API那样清除地理解记录的内部结构，所以往往需要程序员自己写优化代码。

Spark的结构化API会自动以优化后的二进制压缩格式存储数据u，而在使用低级API时，为了实现同样的空间效率和性能，需要在对象内部实现这种压缩格式，以及针对该格式进行计算的所有低级操作。同样，像重排过滤和聚合等这类 SparkSQL 中自动化的优化操作，也需要手动实现。

RDD API 与 Dataset 类似，不同的是 RDD 不在结构化数据引擎上进行存储或处理。然而，在 RDD 和 Dataset 之间来回转换的代价很小，因此可以同时使用两种 API 来取长补短。

2.1. RDD类型

Spark API 文档中，有很多RDD子类，大部分是 DataFrame API 优化物理执行计划时用到的内部表示，而作为用户，一般只会创建两种类型的 RDD:

  • 通用型RDD
  • 提供附加函数的key-value RDD，例如基于key的聚合函数

这两种最常用，都是表示对象的集合，但是 key-value RDD 支持特殊操作并支持按key的自定义数据分片。

现在来正式定义 RDD，每个 RDD 具有以下五个主要内部属性：

  • 数据分片（Partition）列表
  • 作用在每个数据分片的计算函数
  • 描述与其他RDD的依赖关系
  • （可选）为ke-value RDD配置的Partitioner（分片方法，如hash分片）
  • （可选）优先位置列表，根据数据的本地特性，指定了每个Partition分片的处理位置偏好（例如，对于一个HDFS文件来说，这个列表就是每个文件块所在的节点）

如果正确地使用你自定义的Partitioner，则能在性能和稳定性上得到有效提升（将在下一篇博客介绍键值对RDD时，深入介绍Partitioner）。

RDD 支持两种类型的算子操作：惰性执行的转换操作和立即执行的动作操作，都是以分布式的方式来处理数据。

RDD 中没有“行”的概念，RDD 的单个记录只是原始的 Java/Scala/Python 对象，因此必须手动处理这些数据，而不能使用结构化API中的函数库。

对于 Scala和Java 而言，使用RDD的性能基本是相同的，主要的性能开销花费在处理原始对象上；而对于 Python，使用RDD会极大地影响性能（运行 Python RDD 等同于逐行运行用户自定义的 Python函数，我们需要将数据序列化到 Python 进程中，使用 Python 进行操作，然后在将其序列化到Java虚拟机中）。

2.2. 何时使用RDD

除非有非常明确的理由，否则不要手动创建 RDD，在绝大多数情况下，Dataframe 比 RDD 更高效、更稳定并且具有更强大的表达能力。

当你需要对数据的物理分布进行细粒度控制（自定义数据分区）是，可能才需要使用RDD。

2.3. Dataset和使用Case Clase转化的RDD

使用Case Clase转化的RDD 和 Dataset 的不同之处在于，虽然它们可以执行相同的功能，但是 Dataset 可以利用结构化API提供的丰富功能和优化，无需选则在“JVM类型”还是“Spark类型”上进行操作。

你可以采用其中最简单或最灵活的方式。

4. 操作RDD

操作 RDD 的方式与操作 DataFrame 的方式非常相似，核心区别在于 RDD 操作的是原始 Java 或 Scala 对象而非 Spark 类型，以及还缺乏一些可以帮助你简化计算的“辅助”方法或函数，因此必须自己定义每个过滤器filter、映射函数map functions、聚合aggregation，以及其他想被作为函数使用的任何操作。

4.2.3. count

  • countApprox

是count的近似方法，用于返回大概的技术结果，但它必须在指定时间内执行完成（当超过指定时间时，返回一个不准确的近似结果）。

confidence（置信度）表示近似结果的误差空间包括真实值的概率，也就是说，如果以0.9的置信度重复调用countApprox方法，则表示在90%的情况下的近似技术结果中为真实的计数值。

  • countApproxDistinct

计算去重后的值的大约个数。

有两种实现：第一种实现，传入函数的参数是相似精度，值与值之间的相似度达到该参数指定值，则将其看作是一样的值，相似精度值越小，代表值与值之间越相似，计数结果正确性的可能越大；第二种实现，基于两个参数来指定相似精度，一个用于“常规”数据，另一个用于稀疏数据，这两个参数分别是p和sp，p表示准确率（precision），sp表示稀疏准确率（sparse precision），相似精度约为 1.054/sqrt(2^p)，设置非零值sp（sp>p）能减少内存消耗，并在处理选择性数据时提高准确性。

4.2.6. take

take和它的派生方法（takeOrdered, takeSample, top）的功能时从 RDD 中读取一定数量的值：首先扫描一个数据分区，然后根据该分区的实际返回结果的数量来预估还需要再读取多少个分区。

takeSmaple函数用于从RDD中获取指定大小的随机样本，并可以指定withReplacement（采样过程是否允许放回）、返回样本数量和随机数种子这3个参数；按默认排序，top与takeOrdered返回值的排序顺相反。

4.4. 缓存

缓存RDD的原理与缓存DataFrame和Dataset相同，我们可以缓存或持久化RDD。

可以通过 org.apache.spark.storage.StorageLevel 来指定单例对象的任意4存储级别，存储级别包括内存（memory only）、仅磁盘（disk only）、对外内存（off heap）, 默认情况下缓存或持久化到内存。

4.5. 检查点

DataFrame API 中没有检查点（checkpointing）这个概念。

检查点是将RDD保存到磁盘上的操作，以便将来引用此RDD时能直接访问磁盘上的那些中间结果，而不需要从其源头重新计算此RDD，它与缓存类似，但是它不存储在内存中，只存储在磁盘上。

4.6. 通过pipe方法调用系统命令操作RDD

pipe方法可能是Spark最有趣的方法之一，通过pipe方法，可以利用流水线技术调用外部进程来生成RDD。

将每个数据分区交给指定的外部进程来计算得到结果RDD，每个输入分区的所有元素被当作另一个外部进程的标准输入，输入元素由换行符分隔；最终结果由该外部进程的标准输出生成，标准输出的每一行产生输出分区的一个元素。（空分区也会调用一个外部进程）
